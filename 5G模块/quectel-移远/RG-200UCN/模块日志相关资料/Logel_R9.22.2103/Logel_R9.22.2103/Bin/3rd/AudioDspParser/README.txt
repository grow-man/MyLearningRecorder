关于新工具几点说明
、
（1）工具使用
		本工具仅单位内部使用，不可外流。工具可能存在BUG，有待进一步测试验证及优化。

（2）兼容性
 		该版本目前只在whale2平台上测试验证过，不一定支持其他平台（如shark系列）。
		兼容可解析只dump 6个点数据的mem文件，能够解析出WB point1，6数据。 

（3）更新内容
		新增的几点数据的位置
		原先6点，增加4点（point7，8，9，10），其中：
			DL  : audio_dl_process_frame --> point7 -->  NXP/CVS
		
			UL  : audio_ul_process_frame <-- point8 <-- NXP/CVS <--point4 (Main MIC)
			                                               |	| <--point10(Aux MIC)   
			      					                               |
			      					                               | <-- point9(Ref)
			      
		注：如果以后还需要增加dump点，只需要修改DSP code里trace_head参数值（更新点数值和数据大小值）
		  
（4）使用拓展
		如何检查从*.mem文件解析出的pcm文件帧数是否变少，或者是否某包某点数据没有了
		工具解析 *.mem,使用UltraEdit打开生成的out.txt文件,   
 				164( 3)  -   164( 4)  -    46( 1)  -   164( 2)  -   164( 7)  -   164(10)  -   164( 9)  -   164( 8)  -   164( 5)  -    46( 6)  -  
				              |   |                       |               
				              |   |___point               | 
				 data words___|                           |___point2 164 words data
		 
		可以看出point数据顺序，3--4--1--2--7--10--9--8--5--6
		
		“视图”-->“显示行号”,可统计共有多少行，设定这里有 N 行，即有 N 帧数据包。
		对应的TP文件TRACE点：0xd7e8（voiceCall函数第一个TRACE点）的个数应也为 N，则dump 数据没有丢包。
    
    注：out.txt文件可能会出现如下两种情况
    	（a）前几包没有point7，8，9，10 数据
    	     原因：在电话刚建立时，某些参数还没初始化好，voiceCall里没有执行DL_process/UL_process,没有dump出point7，8，9，10
    	（b）最后一包可能没有 顺序3--4--1--2--7--10--9--8--5--6的后面几个点（如point10，8，9，5，6）的数据，如下：
    			 164( 3)  -   164( 4)  -    46( 1)  -   164( 2)  -   164( 7)  -   164( 10)  -     0( 0)  -     0( 0)  -     0( 0)  -     0( 0)
    	     原因：在电话结束时，由于task的结束，后面几包数据被存在了buffer里没被送出，会但在下一次电话继续送出去
    	           也由于这个原因，解析出来的几个pcm文件大小可能存在一包数据的差距（NB：320bytes，WB：640 bytes）
